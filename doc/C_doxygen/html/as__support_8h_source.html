399 /* In order to write OS-independent code, that may alternatively also execute
400  * without any OS at all, the following functionality is required:
401  *
402  * 1. A portable mechanism to run small recurring tasks in the background ("tasklets"),
403  * particularly for bare metall cases without thread support.
404  *
405  * 2. Synchronisation primitives, particularly for the OS case, where concurrent
406  * threads are available.
407  *
408  * 3. Support for virtual memory address translation.
409  */
410 
411 
412 /********** Tasklets and 'as_yield' **********/
413 
414 /* Tasklets are small tasks that need to be executed regularly (e.g. checking for new
415  * data in a buffer and eventually copying it out of the way).
416  *
417  * Callers (applications, module drivers) should regularly call 'as_yield', e.g. inside
418  * waiting loops, to make sure that the tasklets are executed.
419  *
420  * Tasklets are typically executed if 'as_yield' is called. Depending on the OS, they
421  * may also be performed in a background thread. Hence, they must be synchronized
422  * correctly with the foreground thread if 'AS_WITH_MULTIPROCESSING' is
423  * set. In the POSIX backend, the switch 'SERIALIZE_TASKLETS' decides whether tasklets
424  * are synchronized with the main thread internally.
425  */

589  * can (and should) be implemented in a portable way using a condition variable:
590  * The waiting function repeatedly calls 'as_cond_wait', the respective tasklet
591  * in turn calls 'as_cond_signal' if new work is available.

593  * On single-processing systems, this will end up in a busy loop repeatedly
594  * executing 'as_yield ()'. On multi-processing system, the waiting thread will
